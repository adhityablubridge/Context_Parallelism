The core algorithm — the all-to-all layout transform, the claim that combine and partition are exact structural inverses, and the consequence that backward is just the same operations in reverse order — is correct and is the right way to think about it. A combine/partition step is a pure data permutation (no reduction, no duplication), so its backward genuinely is the inverse permutation applied to the gradient with no extra scaling, which is what you've written for `UlyssesAttentionBackward`. The square-causal local SDPA reuse is also sound: once every rank holds the full contiguous sequence `0..T-1` for its head group, `is_causal=true` on the gathered tensor is exactly equivalent to the reference, and since it's a single block there's genuinely no LSE merge needed. So the math is right. The risk in this plan isn't in the math, it's in two places where you're relying on contracts from the ring code that the plan doesn't pin down explicitly enough.

**The `pre_sharded` contract is the biggest risk.** Ulysses' correctness depends entirely on the T-sharding being contiguous in rank order (rank 0 = positions `0..T_local`, rank 1 = next block, etc.) — you call this out explicitly for the `!pre_sharded` path, where you control the sharding yourself via `make_shards_inplace_axis`. But for the `pre_sharded` path you're trusting the caller to have already sharded the sequence the same way. The existing ring path's callers almost certainly hand you HeadTail/zig-zag-sharded input, because that's what ring needs. If anyone calls `forward_cp` with `pre_sharded=true` while in Ulysses mode and the upstream sharding is still zig-zag (easy to do by accident once both modes coexist and a caller just flips `use_ulysses_` without re-checking how it shards its inputs upstream), you get no shape mismatch, no crash — just a silently wrong causal mask and silently wrong gradients, because the concatenated "sequence" inside `combine` no longer corresponds to the real global order. I'd either forbid `pre_sharded=true` for Ulysses for v1, or add a runtime tag/assert distinguishing "contiguous-sharded" from "zig-zag-sharded" tensors so this can't be passed in silently mismatched.

**Reuse of the ring path's unshard/re-shard helpers has the same failure mode.** The plan says to reuse "the same unshard/all-gather helper" for the forward output and the same re-shard logic in backward for `grad_out`. If those helpers are generic (gather along an axis, concatenate in rank order, nothing more), reuse is fine. But if they bake in zig-zag-specific reordering — which is plausible, since HeadTail load balancing usually requires the unshard step to know how to de-interleave the zig-zag blocks back into linear order — then running that same helper on Ulysses' already-contiguous shards would scramble them. This is worth checking directly in `RingRotator.h`/`ContextParallel.h` before wiring it up rather than assuming reuse is safe; if the helper is zig-zag-aware, Ulysses needs its own (much simpler) plain-gather variant instead.

**Test coverage gap at ws=2.** A two-rank parity test can pass even with a transposition bug in how the `P` axis is laid out relative to `B`/`H_local`/`T_local`, because with only two ranks there's only one possible "swap," so certain classes of indexing bugs (e.g., the permute axes in step 2/4 of combine being subtly wrong) are exactly the kind of thing that cancels out or stays correct by symmetry at P=2 but breaks at P=4. I'd add a `ws=4` parity run before treating this as proven correct, and explicitly test all four combinations of `pre_sharded`/`unshard` flags given the risk above, not just the default off/off path.

Two smaller, non-blocking notes: the plan doesn't mention that Ulysses' local SDPA now operates on the full `T×T` causal matrix per call (vs. ring's incrementally-sized blocks), so peak activation memory per rank goes up even though FLOPs/compute are comparable — worth a line in the doc so it doesn't surprise anyone doing capacity planning at large `P` with few heads per rank. And doing three separate all-to-alls for Q/K/V (rather than concatenating along the head or last dim before one call) is fine for correctness in v1 but is an easy later win alongside the already-planned async/overlap work.
