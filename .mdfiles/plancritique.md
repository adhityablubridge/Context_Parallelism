The plan is well-structured and the math in §B is correctly derived and internally consistent with the requirements doc — I checked the monotonicity of `global_pos` across the head/tail split, the sub-chunk "equal-delta is a no-op" claim, and the `s=(r-i) mod N` substitution that lets the `i=0` diagonal fall out of the *same* formula without a special case. The phase sequencing (freeze the interface, then parallelize plumbing + a de-fused oracle, then integrate, then optimize) is the correct call — Phase 2's oracle is genuinely the highest-leverage move here, for a reason worth making explicit below.

There's one real design gap, though, and it's the kind that won't show up until you're running on more than a couple of ranks.

**The gamma-gradient accumulation conflates two different things.** Both docs describe `dq_gamma`/`dk_gamma` as "accumulate across ring steps, like dQ/dK." But `dQ`/`dK` are correct to ring-accumulate because Q and K are *sharded* — each rank owns a disjoint slice of the data, so summing contributions as the shard travels the ring gives you the complete gradient for that shard, full stop. `q_gamma`/`k_gamma` are QK-norm scale parameters, almost certainly shape `[hd]` or `[heads, hd]` — the *same* parameter values used by every rank for every token. They're replicated, not sharded. That means a single rank's full ring traversal only gives you that rank's *partial* contribution to the gradient; the true `dgamma` is the sum over all N ranks' partial sums, which requires a reduction across the CP process group (the same pattern as any replicated parameter in tensor/sequence-parallel training), not just an in-ring accumulator.

This matters because it's exactly the bug that hides from your own test plan: if Phase 2's parity gate compares a single rank's accumulated `dq_gamma`/`dk_gamma` against a slice of the ws=1 reference, it'll look fine in isolation but be silently short by a factor of N once you sum across the real CP group in training. I'd add two things to the plan:
- Phase 1c: an explicit line that `dq_gamma`/`dk_gamma` need a CP-group all-reduce *after* the ring completes, distinct from the within-ring dK/dV-style accumulation — this is new plumbing, not a relabeling of existing dQ/dK code paths.
- Phase 2c: a parity gate that specifically sums the per-rank gamma grads across the simulated ring and checks that sum against the ws=1 single-GPU gradient — not just that each rank's local value is "reasonable."

Two smaller things worth pinning down while you're at it:

The backward signature in Phase 0 says the caller supplies LSE "merged or per-step" per §3.5's lesson, but doesn't say the same about `out`/`dO`. In principle O/dO should always be the *final*, fully-merged values (correct regardless of the partial-step LSE bug, since they're only materialized after the whole ring's forward+merge completes) — only the LSE *fed into the recompute* needs the per-half treatment under HeadTail. Worth a one-line clarification in the Phase 0 contract so it isn't ambiguous later, and worth making sure Phase 2c's "off-diagonal/skip steps" parity wording explicitly includes the load-balanced *partial-Q-half* backward case, not just the literal skip case — they're different failure modes.

Second, if Phase 0 picks the "global mask from deltas" option over the existing zigzag-selection trick (§4.2's bonus), the kernel's diagonal/tile early-exit logic (the `max_kj`-style bound mentioned in §3.2) needs to be re-derived against the piecewise head/tail deltas, not the flat single-offset version. It's a small thing but an easy place for an off-by-one to slip in unnoticed since the existing early-exit math assumes a contiguous offset.

None of this invalidates the approach — the 4-delta interface itself is sound and the parallel-track strategy is the right one. The gamma all-reduce is the one I'd actually block on getting written down before Phase 1 lands, since it's invisible in any single-rank or even small-ring test and only bites in real multi-rank training.
