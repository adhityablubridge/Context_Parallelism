Two things I'd still pin down before calling this closed:

**AVG vs SUM is an assumption, not a given, even with the `:1407` precedent.** Whether the cross-rank combination should be average or sum depends on how each rank normalizes its local loss contribution under CP. If each rank normalizes by the *global* sequence length T (so the per-rank gradients are already directly summable to reconstruct the full-sequence gradient), the correct reduction is SUM. If each rank instead normalizes by its own local shard length T/N (treating its shard like a self-contained mini-batch, DDP-style), then AVG is correct. Matching `:1407` is the right instinct — don't re-derive it from scratch — but it's worth one explicit check: confirm whatever `:1407` reduces is *also* sharded along the sequence dimension within the *same CP group*, normalized the *same way* as gamma's loss path, and not actually a DP-group reduction that happens to share code. If it's a DP-group reduce being reused for a CP-group problem, the group (and possibly the denominator) could be wrong even though the AVG-vs-SUM call ends up right by coincidence. One line in the gate description ("confirmed `:1407` reduces over the CP communicator with matching normalization") would close that off.

**The GAMMA-GRAD gate is in the wrong phase to catch what it's designed to catch.** Phase 2's stand-in applies QK-norm as a standalone op *outside* attention (per the de-fused Option A design) — so `q_gamma`/`k_gamma` there flow through ordinary autograd with no manual per-step accumulation at all. There's no ring-rotation bug, no all-reduce-wiring bug, *no manual code path to be wrong* in that path. Running the GAMMA-GRAD gate against the Phase-2 stand-in will pass trivially regardless of whether the real accumulation logic is correct, because the stand-in doesn't exercise that logic. The gate as written only becomes a real test once the fused kernel's manual `dq_gamma`/`dk_gamma` outputs exist — i.e., it belongs in Phase 3b, not Phase 2c.

That said, you don't have to wait for Phase 3 to catch the wiring bug. I'd add a cheap Phase 1 exit criterion: have the Phase-0 stub kernel return deterministic dummy per-step gamma values (e.g. `step_index + 1`), then unit-test that your local-accumulate + simulated-AVG-across-ranks logic produces the closed-form expected sum — no GPU, no real kernel, no multi-rank job needed. That's the earliest and cheapest point to catch "forgot `register_parameter`," "averaged when I should've summed," or "accidentally rotated it after all" — exactly the class of bug this whole gamma thread started over — well before Phase 3 makes it expensive to debug.
